# HoldemMates 프로젝트 컨텍스트

## 언어 지시 (⚠️ 최우선 — 항상 준수)
- **항상 한국어로 답변할 것.** 코드 주석/커밋 메시지가 아닌 모든 설명, 요약, 대화는 한국어로 작성
- 작업이 길어지거나(대규모 리팩터링, 여러 파일 순회, 긴 도구 호출 연속 등) 컨텍스트가 쌓여도 **중간에 영어로 전환되지 않도록** 매 응답마다 한국어 유지를 재확인할 것
- 사용자가 영어로 질문해도 답변은 한국어로 할 것 (사용자가 명시적으로 다른 언어를 요청하지 않는 한)

## 프로젝트 개요
- 친구들끼리 쓰는 포커(홀덤) 유틸리티 웹앱
- 배포 주소: holdemmates.com (GitHub Pages + Cloudflare DNS)
- GitHub 저장소: Zzang-Ki/Holdem
- 1인 개발, 저녁 시간 짧은 세션(45~60분)으로 반복 작업하는 방식

## 파일 구조
- `index.html` — 정산 계산기 (메인 페이지)
- `pushfold-chart.html` — Push/Fold 차트
- `CNAME` — Cloudflare 커스텀 도메인 설정 (holdemmates.com)
- 별도 프레임워크/빌드 과정 없이 **단일 HTML 파일** 구조 (핵심 기능들이 각각 독립된 HTML 파일)

## 기술 스택 & 배포
- GitHub Pages로 정적 호스팅, 도메인은 Cloudflare에서 구매 후 DNS 연결
- main 브랜치에 푸시하면 GitHub Pages에 자동 반영됨

## 주요 기능

### 1. 정산 계산기 (Settlement Calculator)
- 홀덤 정산 처리 핵심 기능, 단일 HTML 파일
- 바이인 스테퍼, 한국어 단위 표시, 정산(비용 분배) 로직
- html2canvas로 결과를 이미지로 공유
- 라이트 컬러 테마

### 2. 날짜별 방(Room) 구조
- 날짜를 키로 하는 방 아키텍처
- localStorage 캐싱 + 날짜 목록 랜딩 화면

### 3. Push/Fold 차트 (`pushfold-chart.html`)
- Nash 핸드 순서 전체 반영
- 한국어 UI 라벨
- 인원수 / 스택 사이즈 / 앤티 토글
- Push 모드 / Call 모드 각각 동적 힌트 텍스트

## Firebase 연동 (⚠️ 중요)
- Firestore 사용 (`index.html` 내 클라이언트 SDK, compat 버전)
- **소유자 전용 쓰기 권한** — `index.html`의 `HOST_EMAIL` 상수(iamrlgns@gmail.com)로 로그인한 계정만 쓰기 가능
- 그 외 사용자는 **읽기 전용**
- 실제 권한 보장은 클라이언트 코드가 아니라 **Firestore 보안 규칙**에서 이루어짐 — 규칙이 `request.auth.token.email == HOST_EMAIL` 등으로 실제 서버 측에서 막고 있는지 항상 확인
- `firebaseConfig`의 `apiKey` 등 클라이언트 웹 설정값은 원래 공개되어도 되는 값이라 `index.html`에 커밋되어 있음 (정상) — 절대 커밋하면 안 되는 건 **서비스 계정 JSON 키(Admin SDK)** 같은 서버 전용 시크릿이며, 이 프로젝트는 현재 그런 백엔드를 쓰지 않음. 추후 Cloud Functions/Admin SDK를 추가한다면 그 키는 반드시 `.gitignore` + 환경변수로 관리

## 작업 시 유의사항
- 기존 단일 HTML 파일 구조를 임의로 프레임워크화하지 말 것 (명시적 요청 없는 한)
- 새 기능 추가 시 기존 한국어 UI 라벨 스타일 유지
- 변경 후 Firebase 권한 구조(소유자 쓰기 / 읽기 전용)가 깨지지 않는지 확인
- 배포는 main 브랜치 푸시 = 즉시 반영됨 (브랜치/PR 정책은 아래 '작업 규칙' 참고)

## 세션 시작 시
1. 새 작업을 시작하기 전에 항상 git pull로 origin/main과 동기화 상태를 확인할 것.
2. 로컬에 커밋 안 된 변경사항이 있으면 먼저 알려줄 것.

## 작업 규칙
1. 기능 작업 전 specs/에서 해당 스펙을 먼저 읽는다. 스펙이 없으면 구현하지 말고 사용자에게 스펙 작성을 요청한다.
2. 상태가 "확정"인 스펙만 구현 대상이다.
3. 스펙에 없는 범위는 임의로 확장하지 않는다. 필요하면 제안만 남긴다.
4. 구현 후 스펙의 "완료 조건"을 하나씩 직접 검증하고 결과를 보고한다.
5. main에 직접 푸시하지 않는다. feature/기능명 브랜치 → PR로 제출한다. 예외: specs/ 등 문서(.md) 변경은 main에 직접 커밋해도 된다.
6. 완료된 기능은 스펙 상태를 "완료"로 갱신한다.

## 스펙 파일 관리 규칙

### 파일 분리 기준
- 스펙 파일은 기능 단위로 하나씩 만든다.
- "확정 → 개발중 → 완료" 사이클이 한 번 끝난 스펙에 새로운 요구사항이 크게 붙으면, 같은 파일을 계속 늘리지 말고 새 파일로 분리한다.
- 아직 "초안" 상태에서 협의 중인 스펙은 새 파일을 만들지 않고 기존 파일을 계속 수정한다.
- 스펙이 "완료"되어도 파일은 삭제하지 않고 보존한다 (추적 기록 목적).

### 파일명 규칙
- 형식: kebab-case, 영문 소문자 + 하이픈만 사용 (예: `home-grid-redesign.md`)
  - 한글 파일명 금지 (인코딩 이슈 방지). 파일 안의 기능명(H1 제목)은 한국어로 작성
- 구조: `{영역}-{동작/내용}.md`
  - 영역 예시: home, ledger(정산), pushfold, auth(권한) 등
  - 예: `home-grid-redesign.md`, `home-recent-game-card.md`, `auth-user-roles.md`
- 날짜 prefix는 넣지 않는다 (히스토리는 git 커밋이 담당). 같은 영역+내용의 별도 버전을 나란히 남겨야 하는 특수한 경우만 `-v2` 접미사 허용
- 같은 "영역" 접두사를 쓰는 파일들은 서로 연관 기능임을 이름으로 알 수 있게 한다
- 파일명부터 정하고 그 이름에 맞게 스펙 범위를 좁힌다. 파일명과 다른 주제가 섞이면 별도 파일로 분리해야 한다는 신호로 본다
- 언더스코어 prefix(`_template.md` 등)는 실제 스펙이 아닌 메타 파일 전용

### 인덱스 관리
- `specs/roadmap.md`가 전체 스펙 목록 + 상태를 관리하는 인덱스 역할을 겸한다.
- 새 스펙 파일이 생기거나 상태가 바뀌면 `roadmap.md`의 표도 함께 갱신한다.

## 세션 종료 시
1. 작업이 끝나면 변경사항을 커밋하고 push까지 완료했는지 확인할 것. 스펙 문서는 초안(WIP) 상태여도 커밋해서 올려둔다.
2. 커밋하지 않고 남긴 변경사항이 있으면 사용자에게 목록으로 알려줄 것.
